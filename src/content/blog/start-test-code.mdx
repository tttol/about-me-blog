# はじめに
テストコードがない、あるいはテストコードが存在はするがあまり機能していないプロジェクトに途中から参画したとき、あなたならどうしますか？「テストを書きましょう」と言うだけではチームは中々動きません。

テストコードが浸透していないコードベースには必ず理由があります。テストを書く時間がない、テストの必要性が共有されていない、あるいはやむを得ない事情で書いていないなど。まずは既存コードとそれを書いてきたチームに敬意を払い、背景を丁寧にヒアリングすることが出発点です。

本記事ではテストを導入するための技術的なステップはもちろん、チームの合意を得るための対話の進め方を共有します。

※本記事で言及するテストとはJavaのJUnitやTSのVitestなどのテストフレームワークによって実装されるテストコードを指しています。

# テストコードが根付くまでのステップ
テストコードを根付かせるためのステップは大まかに以下の通りです

- 現状分析（スタート）
- テストがない/少ない原因の解明
- 既存テストと向き合う
- 新しいテストを書く
- CIによる自動化は必須
- 小さな成功体験を作る（ゴール）

# 現状分析（スタート）
コードベースをgit cloneし、テストコードの有無を確認。
ある場合、どんなテストを書いているか？UT？IT？モック利用有無は？実行頻度は？カバレッジは？
ない場合、元からずっとないのか、ある時点まであったけど何らかの事情でやめたのか、代わりに何のテストをしているのか（マニュアルテスト？）

まずはコードベースをgitからcloneし、テストコードの実装状況を確認するところから始めます。
テストが全くのゼロの状態の場合、残念ながらコードベースからわかることはあまりありません。テストコードがない代わりに他にどんなテストを行っているのかを確認します。Playwright, Selenium, マニュアルテストなど何かしらのE2Eテストを行っていることが多いはずです。
テストがある場合、そのテストが正しく実装・運用されているかどうかを確認します。これはテストコード自体はあるものの、何かしらの問題点が残っているケースを指します。

実際に私が経験したパターンは以下です
- 一部の機能にだけテストコードがあるが大部分の機能にはテストがない
- テストはあるが実行してみるとエラーになる
- Unit Testは実装されているがIntegration Testは実装されていない
- テストは存在するがCIで自動実行する基盤がない


# テストがない/少ない原因の解明
なぜない・少ないのか
のっぴきならない事情があるかもしれない

現状がわかったら次は原因を考えます。
まず切り分けるポイントとしては、「意図的にテストを書かないという選択肢をとっている」or「テストを書きたいけど何らかの事情で書くことができない」です。

### 意図的にテストを書かない選択肢をとっているケース
このケースで考えられる理由は以下です
- テストを書く時間がない
- テストを書く必要性がわからない

これらの意見に対する自分の回答は「テストを書くことでデグレリスクが低減し結果的にトータルの開発リードタイムが短縮される」です。

# テストの強化を提案する
ここが一番重要かつ慎重に行うべきところ
「テストを書きましょう」と言っても人は動かない
まずは既存コードベースを育てたメンバーに敬意を払い感謝すること。テストを書かない/書けない理由を丁寧にヒアリングしボトルネックの改善を目指す。

現状分析と原因がわかったら、いよいよテストの強化に入ります。
ですが、ここ単にで「テストを書きましょう！」と発言するだけではいけません。ここでどういう言い方をするかが一番重要かつ慎重に振る舞うべきポイントです。

今回のテストの例に限った話ではありませんが、コードベースに対する改善提案を出す場合の前提のマインドとして「既存コードに敬意を払う」というところを自分は大事にしています。現在のコードベースがどんなにひどい状態だったとしても、そのコードベースを何ヶ月・何年も面倒をみてきたメンバーを批判する理由にはなりません。（批判したくなる気持ちはとてもわかります）ビジネスサイドの事情や納期の事情など様々な理由で、ベストプラクティスから逸れた実装をせざるをえないシーンはよくあります。

まずは既存メンバーに対して感謝の気持ちを持ちましょう。

その上で、テストを強化することでさらに改善されるよ！という言い方でアプローチすると良いです。

敬意がないまま改善提案に入ると、人は無意識のうちに攻撃的になる気がします。「ここがイマイチなのでもっとこうしましょう」という高圧的な印象を相手に与えてしまう可能性があります。

正直、改善提案をする際の最適な言い回しはチームやプロジェクトの状況によって異なるので、決まった答えはありません。敬意を持って接しても却下されることもあるでしょう。

### まず感謝から入る
「ありがとう」から入ることを意識している
xxxありがとうございます。
例え相手が何か間違っていたとしても、まずは自分のために時間を割いてくれたことに敬意を払う

### 共感力
テストを書かない・書けない事情に共感する。
子育てでも同じ。ハサミを触りたがるなら「ハサミって光っててかっこいいし触りたいよね。でも手を切るリスクがあるからパパは触って欲しくない。怪我して欲しくないから見るだけにしておこうか」といった感じ。なお毎回通じるわけではない。

# 既存テストと向き合う
UTなのかITなのか
カバレッジはいくつか
自動実行しているか

### カバレッジにこだわりすぎない
85~90%ぐらいを目標に掲げることが多い
100％を目指すのは危険。それ自身が目的になり、本質的ではない。
カバレッジは今あるケースだけをカバーするものであり、考慮漏れのケースはカバーできない。

# 新しいテストを書く
不足しているテストケースがあれば書く。
ゼロから始める場合はど正常系から書き始めるのが良い。異常系は思い切って諦める。
なるべくモックを使わない・実装の詳細に依存しない・ハッピーパスにスコープを絞る

# CIによる自動化は必須
テストコードは自動実行されてこそ輝く
最初の実装コストは高いかもしれないが、リターンは大きい。日々のコミットをトリガーにして勝手にテストが走る恩恵は大きい。

# 小さな成功体験を作る（ゴール）
テストコードを実装・実行することで得られる成功体験を作る
「テストがあってよかった」と思える体験が大事。これがないと日々テストを書くモチベーションが下がる
